# 订单-出货-应收核心业务护栏设计

## 背景

巨人纺织 ERP 现在已经从手工 Excel 记账，逐步变成覆盖销售订单、成品库存、出货、退货、加工单、应收对账和打印单据的业务系统。系统已经能解决日常工作，但最近连续暴露出一个更大的风险：新增或修复一个功能时，已经确认正常的旧规则被意外破坏。

典型问题包括：

1. 快速无订单出货的送货单打印有金额，但进入应收对账后金额变成 0。
2. 已作废送货单曾被纳入应收对账，导致对账单金额和真实出货不一致。
3. 同一客户同一月份需要按订单号拆开对账，但编辑对账时又可能被混合到同一张对账单。
4. 同一批布卷、同一缸号、同一重量、同一日期出现重复送货记录，导致未对账预警误报。
5. 重复送货如果直接作废并回库，会把已经真实出货的库存错误放回仓库。
6. 收款账户字段如果在生成对账单时可随手编辑，容易造成打印内容和账户档案不一致。

这一阶段的目标不是增加更多功能，而是用大型 ERP 的方式，把已经确认的业务规则变成清晰边界、状态流转和自动检查。后续任何优化都必须先保护这些规则。

## 目标

第一阶段聚焦一条最重要的业务链：

```text
销售订单 -> 成品入库 -> 出货/快速出货 -> 退货/重复作废 -> 应收对账 -> 收款记录
```

本设计要达到：

1. 明确每个模块负责什么，不让一个功能随意修改另一个模块的核心数据。
2. 明确哪些记录可以编辑，哪些记录下达、打印、对账或收款后必须锁定。
3. 明确重复出货、作废、退货、对账作废时库存和金额怎么处理。
4. 把已确认的规则写进测试和数据巡检，避免“改 B 坏 A”。
5. 保持现有业务入口和数据，不做破坏性重构。

## 非目标

这一阶段不做这些事情：

1. 不把整个系统重写成 React、Vue 或复杂后端。
2. 不做多公司、多租户、多角色审批流。
3. 不自动批量修改真实业务数据。
4. 不把私人账户设为默认收款账户。
5. 不改变已确认的加工单锁定、染整下达、织厂子单等既有方向。
6. 不把“业务补救操作”伪装成普通作废，避免库存和账款被悄悄改乱。

## 模块边界

### 销售订单

销售订单是客户、布类、颜色、价格、付款方式和订单号的来源。普通出货和普通应收对账应优先引用销售订单。销售订单可以被出货、加工单和对账单读取，但下游单据不应反向随意改写销售订单。

### 成品库存

成品库存是布卷状态的唯一权威来源。每匹布卷只能处于一个明确状态，例如在库、已出货、退货待处理、已返修、已报损或已作废。库存模块负责判断布卷是否可出货，不负责计算客户应收金额。

### 出货单

出货单是客户交付事实的来源。普通出货必须引用销售订单和布卷；快速无订单出货可以没有销售订单，但必须自带客户、布类、颜色、缸号、数量、单价和金额计算所需字段。

### 退货和异常处理

退货、重复作废、客户扣减和报损都是异常业务事件，不能混成同一个“作废”。它们必须保留原因和处理方式：

1. 普通作废：真实未出货或操作录错，可以恢复库存。
2. 重复作废：系统重复录入同一批布卷，不恢复库存，只让重复单不进入对账。
3. 退货：客户退回实物，进入退货待处理，不直接变成正常库存。
4. 扣减/附加费：影响应收金额，但不影响库存状态。

### 应收对账

应收对账只引用有效出货记录。它可以汇总普通出货和快速无订单出货，但不能引用已作废、重复作废或无效金额的记录。对账单保存后应保留打印快照，包括当时选择的收款账户、付款方式、到期日、明细和金额。

### 收款账户档案

收款账户应在账户档案中维护，对账单只选择账户并保存快照。生成或编辑对账单时，账户类型、账户名称、开户银行和账号应只读显示，避免误点改乱。

私人代收账户可以存在，但必须符合两个原则：

1. 不作为默认账户。
2. 打印时显示明确提示：个人代收账户需要财务负责人确认，并按公司要求入账申报。

## 状态流转

### 送货单状态

送货单应至少区分这些状态：

1. 有效：真实出货，可打印、可纳入对账。
2. 已对账：已进入某张应收对账单，不应再随意编辑核心数量、价格和客户。
3. 普通作废：记录无效，相关布卷可以恢复为在库。
4. 重复作废：记录无效，但不恢复库存，必须关联保留单据或重复原因。
5. 已退货或部分退货：保留出货事实，同时记录退货处理。

### 布卷状态

布卷状态必须和送货单保持一致：

1. 在库布卷不能关联有效送货单。
2. 已出货布卷必须关联有效送货单。
3. 被普通作废释放的布卷可以回到在库。
4. 被重复作废处理的送货单不能释放布卷。
5. 退货布卷进入退货待处理，不自动进入正常库存。

### 对账单状态

对账单应至少区分：

1. 未结清：已生成对账单，但未完全收款。
2. 已结清：收款金额覆盖应收金额。
3. 已作废：对账单无效，释放出货单的对账关联，但不改动库存。

## 关键业务规则

### 送货唯一性

同一批布卷不能同时存在于多张有效送货单中。系统发现同一 `rollIds` 被多个有效送货单占用时，应提示重复占用，并要求人工选择哪一张保留、哪一张走“重复作废”。不能把重复记录简单普通作废并回库。

快速无订单出货如果没有 `rollIds`，也要用客户、日期、布类、颜色、缸号、数量、单价等字段做重复风险提示。该提示先作为风险预警，不自动阻止真实业务。

### 作废规则

普通作废只适用于真实没有出货或录入错误的单据。普通作废可以恢复库存。

重复作废适用于重复录入同一实物出货的单据。重复作废只让重复单不进入对账和统计，不恢复库存。

已经进入应收对账的送货单不能直接普通作废，必须先作废或调整对账单。

### 应收纳入规则

应收对账只能纳入：

1. 有效普通送货单。
2. 有效快速无订单送货单。
3. 金额可计算且不是 0 的收费样品出货。

应收对账必须排除：

1. 普通作废送货单。
2. 重复作废送货单。
3. 已退货且全额冲销的记录。
4. 缺少客户或金额依据的异常记录。

### 按订单拆分规则

同一客户同一月份的出货，可以按订单号拆开生成多张对账单。这是已确认规则，后续修改不得取消。

对账单编辑界面拉取未对账出货时，默认只显示当前对账单所属订单范围内的记录；如果业务需要跨订单合并，必须是明确的人工选择，而不是系统自动混合。

快速无订单出货没有销售订单号时，应独立归入“无订单出货”组，不能被错误合并到普通订单对账单中。

### 金额计算规则

送货单打印金额、应收对账金额、Excel 导出金额必须使用同一套计算规则。

普通销售订单出货优先使用销售订单里的单价和计价单位。快速无订单出货使用自身记录的单价、米数、公斤数或人工金额字段。收费样品不能因为没有销售订单而变成 0 元。

### 收款账户规则

对账单选择收款账户时，从账户档案读取账户类型、账户名称、开户银行和账号，并保存到账单快照。生成后这些字段只读展示。

打印对账单时使用对账单快照，不直接读取当前默认账户，避免历史对账单因为账户档案修改而变化。

## 数据巡检

`npm run guard` 应持续包含真实数据只读巡检。第一批核心巡检规则如下：

1. 同一 `rollId` 出现在多张有效送货单中。
2. 送货单状态为有效，但引用的布卷不存在。
3. 布卷状态为在库，但仍关联有效送货单。
4. 布卷状态为已出货，但没有有效送货单。
5. 已作废或重复作废送货单仍出现在应收对账单中。
6. 快速无订单收费出货进入应收时金额为 0。
7. 同一客户同一月份不同订单号被自动混入同一张对账单。
8. 对账单缺少收款账户快照。
9. 私人收款账户被设为默认账户。

巡检只报告异常，不自动修改真实数据。需要修复真实数据时，系统应提供明确的人工修正路径和记录原因。

## 测试策略

每条已确认业务规则都要进入测试。后续任何涉及订单、库存、出货、退货、对账、收款账户的改动，都必须先写或更新测试，再改业务逻辑。

测试分成四类：

1. 回归流程测试：验证订单、入库、出货、退货、对账的主链路仍正常。
2. 数据巡检测试：验证重复送货、错误作废、无效对账等异常能被发现。
3. 打印内容测试：验证送货单、加工单、对账单打印内容和金额正确。
4. 账户快照测试：验证对账单使用保存时的账户快照，且私人账户不默认。

统一入口仍然是：

```text
npm run guard
```

通过标准：

1. 所有测试通过。
2. 数据巡检没有新增高风险异常。
3. 修改金额或库存逻辑时，必须能指出对应测试名称。

## 实施顺序

第一阶段建议按以下顺序实施：

1. 建立“业务规则登记表”，把上面的规则和对应测试名称绑定。
2. 集中整理金额计算函数，让送货单、对账单、Excel 使用同一来源。
3. 集中整理送货单状态判断函数，区分有效、普通作废、重复作废、已对账。
4. 加强数据巡检，先发现重复占用、错误对账和金额为 0 的收费样品。
5. 增加人工修正路径，用“重复作废不回库”处理真实重复单据。
6. 固化对账单按订单拆分和无订单出货独立分组。
7. 固化收款账户档案和对账单只读快照。

每一步都必须小提交、小验证，避免一次改太多造成新的混乱。

## 成功标准

这一阶段完成后，应满足：

1. 快速无订单收费样品可以正确进入应收对账，金额不为 0。
2. 已作废和重复作废送货单不会进入应收对账。
3. 同一客户同一月份可按订单号分开对账。
4. 无订单出货不会被错误合并到普通订单对账单。
5. 同一批布卷重复出货会被巡检发现，并有不回库的重复作废路径。
6. 对账单收款账户来自账户档案，生成后只读并保存快照。
7. 私人收款账户可选但不默认，打印时有合规提示。
8. 后续每次优化前后运行 `npm run guard`，能保护已经确认 OK 的业务规则。

## 用户确认点

进入实施计划前，需要用户确认：

1. 本阶段只保护“订单-库存-出货-应收”主链路，不扩展到全部 ERP 模块。
2. 重复单据修正采用“保留真实有效单 + 重复作废不回库”的原则。
3. 同月份对账默认按订单号隔离，无订单出货单独分组。
4. 收款账户统一从账户档案维护，对账单中只读显示并保存快照。
