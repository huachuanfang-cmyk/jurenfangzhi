# 巨人纺织 ERP 第一阶段护栏设计

## 背景

当前系统是一个单页 ERP，主要代码集中在 `index.html`，数据通过浏览器本地缓存和 Supabase 同步。系统已经覆盖销售订单、加工、入库、出货、退货、应收应付、报价和打印等核心业务。

现在最大的风险不是缺功能，而是回归：优化 B 后，已经确认正常的 A 被悄悄破坏，几天后才在真实业务数据里发现。

## 第一阶段目标

第一阶段只做护栏，不做大重构：

1. 增加测试模式，测试数据不得污染真实业务数据。
2. 建立核心流程回归测试，改功能后能快速检查旧流程是否仍正常。
3. 增加数据巡检，发现真实数据中的不一致关系。
4. 保持当前单人使用方式，不引入多用户、复杂后端或大型框架。

## 非目标

第一阶段不做这些事情：

1. 不重写 UI。
2. 不迁移到 React、Vue 或大型前端框架。
3. 不改变日常访问入口。
4. 不改造为多公司、多租户或多用户系统。
5. 不自动修复真实业务数据，只提示异常。

## 方案选择

推荐方案：在现有单页系统旁边加一套最小测试和巡检工具。

理由：

1. 对现有业务影响最小。
2. 能最快解决“改 B 坏 A”的问题。
3. 后续如果要拆模块，也有测试保护。
4. 适合个人 ERP 的节奏，不会变笨重。

## 测试模式

测试模式通过 URL 参数开启：

```text
index.html?test=1
```

测试模式要求：

1. 使用独立的测试数据前缀，例如 `gjr_test_`。
2. 默认不向真实 Supabase 写入数据。
3. 可以在页面内创建测试订单、测试入库、测试出货、测试退货。
4. 测试结束后可以清空测试数据。
5. 正常模式仍使用现有 `gjr5_` 数据前缀。

## 核心回归测试

第一批只覆盖最容易造成损失的流程：

1. 销售订单保存后，订单数据可被读取。
2. 修改订单颜色后，依赖订单颜色的加工单读取到新值。
3. 成品入库后，布卷记录生成，库存状态为在库。
4. 出货后，布卷状态变为已出货，并关联出货单。
5. 退货后，布卷进入退货待处理，不混入正常库存。
6. 应收对账引用出货单后，出货单能标记为已对账。
7. 删除记录时，删除意图必须能同步到数据层，避免云端旧数据复活。

## 数据巡检

数据巡检读取真实数据，但只报告，不自动修改。

第一批巡检规则：

1. 出货单引用了不存在的布卷。
2. 布卷状态为已出货，但缺少出货单 ID。
3. 布卷状态为在库，但仍关联出货单 ID。
4. 退货记录引用了不存在的出货单。
5. 应收对账引用了不存在的出货单。
6. 前端 `TABLE_MAP` 中使用的表，在主 schema 中缺失。
7. 本地存在长时间未同步的 dirty 标记。

## 使用方式

后续每次改功能前后，都跑同一套检查：

```text
npm run guard
```

预期输出：

1. 回归测试通过或失败清单。
2. 数据巡检异常清单。
3. 如果失败，指出是哪条业务链路断了。

## 文件规划

第一阶段新增文件：

1. `tools/guard/erp-core.js`：从页面逻辑中抽取可测试的数据操作模型。
2. `tools/guard/regression-tests.mjs`：核心流程回归测试。
3. `tools/guard/data-audit.mjs`：真实数据巡检。
4. `tools/guard/run-guard.mjs`：统一运行入口。

计划小幅修改：

1. `package.json`：增加 `npm run guard`。
2. `index.html`：仅在必要时增加极少量测试模式检测或数据层导出能力。

## 成功标准

第一阶段完成后：

1. 不需要人工点完整套流程，也能知道核心链路有没有坏。
2. 测试不会污染真实业务数据。
3. 数据巡检能指出潜在异常，而不是等几天后人工发现。
4. 后续每次让大模型优化功能，都能先跑护栏再交付。
